# PrismHR Global Partner-Motion App — Codex Canon & Working Hypotheses

## Purpose

This document tells Codex what to treat as strategic canon versus what to treat as a working hypothesis while helping design or build the PrismHR Global partner-motion command center.

Codex should use this document alongside:

- `deep-research-report.md`
- `updated-master-codex-prompt.md`
- any future internal notes, deck excerpts, CSM interview notes, or product/pricing rules

This document is not a product spec by itself. It is the operating doctrine layer. It tells Codex what must not be misunderstood.

# Part 1 — Treat As Canon

## 1. This Is a Relationship-Owned Channel Motion

This is not a normal direct-sales motion.

This is a relationship-owned channel motion where the scarce asset is **borrowed trust**.

The app must be designed around this reality:

- CSMs own relationship context.
- CSMs own account politics.
- CSMs own intro timing.
- CSMs own permission boundaries.
- Every introduction spends political capital.
- Every sloppy message creates cleanup debt.
- Every premature ask can damage trust.

The product should prevent the user from becoming a liability inside someone else’s trusted channel.

---

## 2. The CSM Is Not A Lead Source

The app must not model CSMs as simple referral machines.

A CSM is a relationship owner. The app must help the user understand:

- what the CSM protects
- what the CSM fears
- what would make the CSM feel exposed
- what makes the CSM trust the user
- what makes the CSM willing to make an introduction
- what makes the CSM avoid making an introduction

The app should help the user become the safest, sharpest, most useful specialist a CSM can bring into a PEO relationship.

---

## 3. Role Definition

The user’s role should be treated as a hybrid of:

- specialist seller
- overlay seller
- partner operator
- motion designer
- rules-of-engagement architect
- discovery and use-case specialist
- channel-intelligence operator

Revenue matters, but the app must also track the softer operating assets that ordinary CRMs miss:

- CSM trust
- PEO protectiveness
- permission status
- intro readiness
- boundary risk
- usefulness of daily serves
- relationship heat
- opportunity momentum
- internal unknowns

---

## 4. Product Philosophy

Do not build a generic CRM.

Do not build an activity tracker pretending to be intelligence.

Build a **partner-motion command center** that makes invisible operating dynamics visible.

The dashboard may be visually expressive or decorative when that treatment improves hierarchy, scanability, product identity, or operator focus. Decoration must not replace evidence, permission, boundary, HML priority, relationship motion, source confidence, unknowns, or next safest action.

Do not expose scaffold, architecture, or implementation status as product UI.

User-facing screens must not explain that the app has a code gate, database connection, owner-write model, server action, build phase, or other developer concern. Those facts belong in code, docs, deployment checks, or admin diagnostics only. The interface should show the operator's work: evidence, permission, boundary, HML priority, relationship motion, source confidence, unknowns, and next safest action.

The app should help answer:

- What matters now?
- Why does it matter?
- What changed?
- What is heating up?
- What is going cold?
- What requires caution?
- What promise is overdue?
- What boundary could be violated?
- Which CSM can the user safely move with?
- Which CSM needs more trust-building?
- Which PEO is ready?
- Which PEO is protected?
- Which account has real global workforce signal?
- Which external channel has actual timing, not just surface area?
- What should the user do next?

For the first real operating surface, treat this as canon:

- The product/information architecture winner is **Prospecting Desk**.
- The visual composition winner is **Relationship Brief**.
- Chicagoland prospecting must be unmistakably central, but the screen should visually lead with borrowed trust, permission posture, source confidence, and next safest action before any action is taken.
- Do not turn this into a generic CRM table. Do not turn it into a decorative trust dashboard that hides the account list.

---

## 5. Permission Is A First-Class Concept

Every account, PEO, PEO client, opportunity, and channel should have a permission state.

Suggested permission states:

- Research only
- CSM context needed
- CSM approved for discussion
- CSM approved for intro
- PEO engaged
- PEO client engaged
- Direct contact allowed
- Direct contact not allowed
- Hold / sensitive
- Off-limits

Permission status must not be buried in notes.

---

## 6. Boundary / Off-Limits Rules Are First-Class Objects

Off-limits rules must be structured.

Track:

- who is off-limits
- what is off-limits
- when it is off-limits
- why it is off-limits
- who set the boundary
- whether it is temporary or permanent
- whether it applies to a CSM, PEO, client, geography, product, channel, or timing window
- what action is allowed instead
- when it should be reviewed again

A clarified boundary is valuable intelligence, not merely a blocker.

---

## 7. Trust Surface Is A First-Class Concept

The app should help protect the CSM’s trust surface.

Track:

- CSM protectiveness level
- CSM communication style
- preferred intro motion
- preferred follow-up motion
- messaging sensitivity
- relationship cadence
- relationship tone
- do’s and don’ts
- what makes this CSM feel supported
- what makes this CSM feel exposed
- whether the CSM wants to stay in every thread
- whether private debriefs are required before external follow-up

---

## 8. Daily Serve Is A Trust-Building Ritual

A daily serve is not a checkbox task.

A daily serve is something useful the user provides to a CSM that makes the CSM’s job easier while also advancing the PrismHR Global motion.

Track:

- CSM served
- PEO or account context
- serve category
- content
- why it is useful
- whether it was sent
- whether it was used
- whether it was forwarded
- whether it generated a reply
- whether it created a next step
- whether it improved relationship heat

The app should reward usefulness, not volume.

A single forwarded daily serve that creates a qualified conversation is worth more than ten logged activities.

---

## 9. HML Signal Meter Is Interpreted Signal

HML means **High / Medium / Low**.

HML is not HTML.

HML is not activity volume.

HML is a living signal layer that reads across the app and turns scattered inputs into prioritized intelligence.

The HML Signal Meter should classify:

- CSM relationship heat
- PEO readiness
- PEO protectiveness / caution
- opportunity momentum
- external channel quality
- Chicago territory account potential
- follow-up urgency
- boundary / off-limits risk
- product/use-case frequency
- country/regulatory complexity
- internal ambiguity risk

Every HML label must explain itself.

Examples:

- “High because this CSM has clear boundaries, responded quickly, reused a daily serve, and approved a PEO intro.”
- “Medium because this PEO appears relevant, but timing is vague and no client use case has surfaced.”
- “High caution because the CSM requested to stay in every thread and flagged the account as sensitive.”
- “Low because there is no visible global hiring need, no recent activity, and unclear ownership.”

No black-box scoring.

---

## 10. Fact / Inference / Unknown Must Be Explicit

The app must distinguish:

- confirmed fact
- public research finding
- CSM-provided context
- internal team-provided context
- reasonable inference
- open question
- unverified assumption

This is mandatory because the motion is still being built.

Inferred information must never appear as confirmed truth.

---

## 11. Internal Unknowns Are Operating Risks

The app should track unresolved internal questions as operating risks.

Examples:

- What exactly is PrismHR Global versus Vensure Global?
- Who owns pricing authority?
- Who owns exceptions?
- Who owns opportunity records?
- Who can contact whom directly?
- What is formally off-limits?
- What does Contractor Management+ include?
- What data can be stored in the app?
- Which notes require restricted visibility?

An internal unknown should have:

- question
- category
- risk level
- owner
- source
- current best answer
- confidence level
- next verification step
- date last reviewed
- resolution status

---

## 12. Rules Of Engagement Must Be Preserved

The app should operationalize this working rules-of-engagement model:

- The CSM owns relationship context, account politics, intro timing, and permission boundaries.
- The user owns specialist diagnosis, use-case framing, product mapping, and recommended next-step design.
- Before a meeting, agree on purpose, attendees, what not to say, and who speaks first.
- During the meeting, let the CSM frame the relationship, then earn the right to lead discovery.
- After the meeting, send the CSM a private debrief first unless direct follow-up has been explicitly delegated.
- Never widen the circle without confirming whether the circle is actually yours to widen.

---

## 13. V1 Signal Engine Should Be Rules-Based

For v1, build the HML engine as a rules-based system.

It should read:

- structured fields
- timestamps
- relationship status
- activity history
- notes
- next steps
- off-limits rules
- CSM sentiment
- PEO readiness
- channel movement
- follow-up due dates
- product/use-case tags
- country/regulatory tags

Later, the architecture should allow AI-assisted summaries and recommendations, but v1 must remain explainable.

# Part 2 — Not Yet Canon / Working Hypotheses

The following items must be treated as working hypotheses until verified internally.

Codex should model these as unknowns, configurable settings, or editable fields — not hard-coded facts.

## 1. PrismHR Global vs. Vensure Global Boundaries

Not yet canon:

- exact commercial boundary
- exact territory/purview boundary
- exact contracting model
- exact branding model
- exact support/service ownership
- escalation path
- who prices
- who approves exceptions
- who owns the record of truth

Build the app so this can be updated.

---

## 2. Contractor Management+ Packaging

Not yet canon:

- whether it means contractor-of-record
- whether it means enhanced compliance
- whether it means indemnity
- whether it means added operational support
- whether it has different pricing
- whether it has different qualification criteria
- whether it has country-specific availability limits

Treat Contractor Management+ as a product object with editable definition, fields, qualification criteria, and discovery framework.

---

## 3. Pricing, Legal, And Approval Rules

Not yet canon:

- pricing authority
- discount authority
- approval flow
- legal review process
- contract owner
- data-processing requirements
- country-level exceptions
- product eligibility constraints

Do not hard-code pricing or approval logic without confirmed inputs.

---

## 4. Direct Contact Rules

Not yet canon:

- whether the user can directly contact PEOs
- whether the user can directly contact PEO clients
- when direct contact is allowed
- who must be copied
- whether CSM approval is always required
- whether certain PEOs are never directly contactable
- whether external-channel accounts have different rules

This should be handled through permission states and off-limits rules.

---

## 5. External Channel Rules

Not yet canon:

- whether external channels are fully approved
- whether non-CSM-associated PEOs can be pursued directly
- whether external channel outreach needs internal approval
- whether external channel leads create conflict with CSM-owned relationships
- whether Chicago Global Chamber has any special status
- whether there are required disclaimers, talk tracks, or boundaries

Build external channels as configurable source records with conflict-risk fields.

---

## 6. CSM Relationship Norms

Not yet canon:

- which CSMs are open
- which CSMs are protective
- which CSMs prefer prep calls
- which CSMs want private debriefs
- which CSMs will allow direct follow-up
- which CSMs want all messaging pre-approved
- which CSMs are comfortable with external-channel discussions

These must be learned and documented.

---

## 7. HML Scoring Weights

Canon:

- HML must exist.
- HML must be explainable.
- HML must interpret signal rather than activity volume.

Not yet canon:

- exact scoring weights
- exact thresholds
- which events contribute how much
- how quickly signal decays
- which fields override scoring
- how caution interacts with opportunity potential

Build HML rules so they are editable.

---

## 8. Daily Serve Cadence

Canon:

- daily serve is a trust-building ritual.
- daily serve should be useful to the CSM.

Not yet canon:

- whether every CSM should receive something daily
- whether “daily” means daily per user or per active CSM
- how often a given CSM wants material
- which serve types are welcome
- which serve types feel like homework

Build cadence as a preference per CSM.

---

## 9. Chicago Territory Prioritization

Canon:

- Chicago is a priority.
- territory scoring should consider visible international activity, contractor intensity, hiring velocity, compliance complexity, channel accessibility, and boundary risk.

Not yet canon:

- final target-account list
- exact scoring weights
- which industries are highest priority internally
- whether certain accounts are already owned
- whether certain categories are off-limits
- whether Vensure Global or another team has existing claims

Build territory records with source confidence, owner, and boundary-risk fields.

---

## 10. Data Visibility / Note Sensitivity

Not yet canon:

- who can see the app
- whether CSMs will eventually use it
- whether managers will use it
- which notes are private
- whether sensitive relationship notes can be stored
- whether customer names or PEO client details can be stored
- retention requirements

Build v1 with conservative note sensitivity and avoid assuming broad visibility.

# Part 3 — Codex Behavior Requirements

Codex must follow these rules when using this canon document.

## 1. Do Not Overbuild

Prioritize the smallest credible MVP that protects the trust-and-signal layer.

Initial priority:

1. CSM Partner Rooms
2. PEO records
3. Permission states
4. Off-limits rules
5. Daily serve tracking
6. HML signal feed
7. Follow-up promises
8. Pitch column
9. Discovery frameworks
10. Internal unknowns tracker

Do not prioritize maps, AI summaries, or forecasting before these basics work.

---

## 2. Always Separate Confirmed From Unconfirmed

When generating specs, schemas, UI, seed data, or sample content, Codex must label items as:

- confirmed
- assumed
- inferred
- placeholder
- needs internal verification

No inferred item should be written as if it is confirmed.

---

## 3. Build For Explainability

Every HML classification must include:

- classification
- reason
- contributing signals
- last updated timestamp
- recommended next action
- confidence level
- source references or related records

---

## 4. Design Around Permission Before Motion

Do not allow workflows that jump straight from “interesting account” to “contact account.”

Every motion should pass through permission/context logic:

1. Research
2. Context needed
3. Boundary check
4. CSM alignment
5. Approved next action
6. Action taken
7. Private debrief / follow-up

---

## 5. Treat The Deep Research Report As Background, Not Gospel

`deep-research-report.md` is a high-value field manual, but it contains public research, industry norms, and reasonable inferences.

Codex should use it to understand the environment, but any internal operational rule must remain editable unless confirmed.
